# Every Minton - 프로젝트 계획서

## 1. 개요

### 서비스 명
Every Minton

### 목적
배드민턴 모임에서 운동 일정을 진행할 때, 모임원들의 **시간당 게임 횟수**가 최대한 균등하게 유지되도록 게임(매칭)을 자동 생성하여 일정 운영을 도와주는 서비스.

### 핵심 지표
```
시간당 게임 횟수 = 진행한 게임 횟수 / 참여 시간(분) × 60
```
- 참여 시간: '대기중' 상태로 전환된 시점부터 현재까지의 경과 시간

## 2. 핵심 기능

### 2.1 일정 관리

| 항목      | 설명                         |
| ------- | -------------------------- |
| 일정 생성   | 날짜, 시작/종료 시간, 코트 수, 장소명 입력 |
| 코트 수 변경 | 일정 진행 중 코트 수 증감 가능         |
| 일정 종료   | 운영진이 수동으로 일정 종료 처리         |

### 2.2 모임원 관리

#### 모임원 등록 (서버에 영구 저장)
- 이름, 성별, 급수(실력) 등록
- 한번 등록된 모임원은 추후 일정에서 이름 검색으로 빠르게 추가 가능

#### 참여 상태 관리
```
참여 예정 → 대기중 ↔ 게임중 → 퇴장
```
| 상태 | 설명 |
|------|------|
| 참여 예정 | 일정에 등록되었으나 아직 도착하지 않은 상태 |
| 대기중 | 운영진이 수동 전환. 이 시점부터 참여 시간 측정 시작. 게임 배정 대기 |
| 게임중 | 게임에 배정되어 플레이 중. 게임 완료 시 자동으로 '대기중'으로 복귀 |
| 퇴장 | 운영진이 수동 전환. 참여 시간 측정 종료 |

### 2.3 게임(매칭) 관리

#### 게임 형태
- **복식(2v2) 전용** (단식은 추후 확장 시 고려)
- 한 게임은 반드시 4명으로 구성
- 가능한 조합: [남2 여2], [남3 여1], [남1 여3], [남4], [여4]

#### 자동 게임 생성
- 기본 동작: 현재 **대기중인 인원**으로 코트 수만큼 게임을 한번에 생성
- 옵션:
  - **게임중인 인원 포함 여부**: 체크 시 현재 게임중인 인원도 매칭 대상에 포함
  - **생성할 게임 수**: 기본값은 빈 코트 수, 수동으로 조정 가능
- 대기 인원이 4명 미만이면 게임 생성 불가

#### 자동 매칭 알고리즘 우선순위
| 순위 (기본값) | 기준 | 설명 |
|---------------|------|------|
| 1 | 시간당 게임 횟수 균등화 | 게임 횟수가 적은 사람 우선 배정 |
| 2 | 같은 조합 반복 방지 | 최근 같이 한 조합을 피함 |
| 3 | 성별 밸런스 | 가능하면 [남2 여2] 우선, 여의치 않으면 [남3 여1] 등 허용 |
| 4 | 급수(실력) 밸런스 | 팀 간 실력 차이 최소화 |

- 운영진이 화면 내에서 우선순위 순서를 드래그 또는 조작하여 변경 가능
- 변경된 우선순위는 해당 일정 내에서 유지되며, 다음 자동 매칭부터 적용

#### 수동 게임 생성
- 운영진이 직접 4명을 선택하여 게임 생성

#### 게임 상태 관리
```
대기 → 진행중 → 완료
```
| 상태 | 설명 |
|------|------|
| 대기 | 게임 생성 직후 상태. 아직 코트에서 시작되지 않음. 배정된 모임원의 상태는 변경되지 않음 |
| 진행중 | 운영진이 수동으로 전환. 이 시점에 배정된 모임원들이 '게임중' 상태로 전환됨 |
| 완료 | 운영진이 "종료" 버튼으로 전환. 해당 4명은 자동으로 모임원 '대기중' 상태로 복귀 |

- 게임 생성(자동/수동) 시 '대기' 상태로 생성
- 운영진이 '진행중'으로 전환해야 게임 시작 (이 시점부터 게임 시간 측정, 모임원 '게임중' 전환)
- 운영진이 '완료'로 전환하면 게임 종료 (모임원 '대기중' 복귀)

### 2.4 대시보드 (실시간 현황)

운영진에게 보여질 정보:

| 정보 | 설명 |
|------|------|
| 시간당 게임 횟수 | 각 모임원의 현재 시간당 게임 횟수 |
| 대기 시간 | 현재시간 - 마지막 게임 종료 시간 |
| 총 게임 수 | 해당 일정에서 진행한 총 게임 수 |
| 모임원 정보 | 이름, 성별, 급수, 현재 상태 |
| 코트 현황 | 각 코트별 진행중인 게임 정보 |

## 3. 기술 스택

### 프론트엔드
| 항목 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | Next.js (React) | AI 코딩 지원 최적, Firebase 통합 용이, 자료 풍부 |
| 언어 | TypeScript | 타입 안정성, AI가 더 정확한 코드 생성 |
| 스타일링 | Tailwind CSS | 빠른 UI 개발, 별도 디자인 시스템 없이도 깔끔한 결과물 |
| 상태관리 | React 내장 (useState/useContext) | 규모가 크지 않아 별도 라이브러리 불필요 |

### 백엔드
| 항목 | 선택 | 이유 |
|------|------|------|
| BaaS | Firebase | 서버 관리 불필요, 실시간 동기화 지원 |
| DB | Firestore | 실시간 리스너로 대시보드 즉시 반영 |
| 호스팅 | Firebase Hosting | 배포 간편, Next.js 지원 |

> **주의**: 백엔드 프레임워크는 추후 변경될 수 있으므로, 백엔드 관련 로직(데이터 접근, 실시간 리스너 등)은 인터페이스로 추상화하여 구현체를 교체할 수 있도록 설계한다.

## 4. 데이터 모델 (Firestore)

### members (모임원)
```
/members/{memberId}
{
  id: string
  name: string
  gender: "male" | "female"
  level: number          // 내부 점수 (1~100)
  createdAt: timestamp
}
```

### schedules (일정)
```
/schedules/{scheduleId}
{
  id: string
  date: string           // "2025-01-15"
  startTime: string      // "19:00"
  endTime: string        // "22:00"
  courtCount: number     // 현재 사용 가능한 코트 수
  location: string       // 장소명
  status: "upcoming" | "in_progress" | "completed"
  createdAt: timestamp
}
```

### participants (일정 참여자)
```
/schedules/{scheduleId}/participants/{memberId}
{
  memberId: string
  status: "registered" | "waiting" | "playing" | "left"  // 참여예정 | 대기중 | 게임중 | 퇴장
  joinedAt: timestamp | null     // '대기중' 전환 시점
  leftAt: timestamp | null       // '퇴장' 전환 시점
  gamesPlayed: number            // 진행한 게임 수
  lastGameEndedAt: timestamp | null  // 마지막 게임 종료 시점
}
```

### games (게임)
```
/schedules/{scheduleId}/games/{gameId}
{
  id: string
  courtNumber: number
  status: "waiting" | "in_progress" | "completed"
  team1: [memberId, memberId]
  team2: [memberId, memberId]
  startedAt: timestamp
  endedAt: timestamp | null
}
```

## 5. 급수(실력) 체계

### 구조
- **내부**: 1~100 숫자 점수로 관리
- **표시**: A~E 등급 + 상/중/하 세분화 (예: "B중", "C상")
- **입력**: 운영진이 등급(A~E) + 세분화(상/중/하)를 선택하면 내부 점수 자동 부여
- **변경**: 운영진이 수동으로만 변경 (자동 변동 없음)

### 등급 ↔ 점수 매핑

| 등급 | 점수 범위 | 범위 크기 | 하 (20%) | 중 (50%) | 상 (90%) |
|------|-----------|-----------|----------|----------|----------|
| A | 90~100 | 11 | 92 | 95 | 99 |
| B | 70~89 | 20 | 74 | 80 | 88 |
| C | 40~69 | 30 | 46 | 55 | 67 |
| D | 10~39 | 30 | 16 | 25 | 37 |
| E | 1~9 | 9 | 3 | 5 | 9 |

### 계산 공식
```
점수 = 등급 최소값 + floor(등급 범위 크기 × 퍼센트)
- 하: 20%
- 중: 50%
- 상: 90%
```

예시:
- B하 = 70 + floor(20 × 0.20) = 74
- B중 = 70 + floor(20 × 0.50) = 80
- B상 = 70 + floor(20 × 0.90) = 88

## 6. 화면 구성 (페이지)

| 페이지 | 경로 | 설명 |
|--------|------|------|
| 홈 | `/` | 일정 목록, 일정 생성 |
| 일정 상세 | `/schedule/{id}` | 대시보드, 참여자 관리, 게임 관리 통합 |
| 모임원 관리 | `/members` | 모임원 등록/수정/삭제 |

### 일정 상세 페이지 구성 (핵심 페이지)
- **코트 현황 영역**: 각 코트별 진행중인 게임과 참여자 표시
- **대기 인원 영역**: 대기중인 모임원 목록 + 시간당 게임 횟수/대기시간 표시
- **게임 생성 버튼**: 자동/수동 게임 생성
- **참여자 관리 영역**: 상태 전환 (참여예정 → 대기중 → 퇴장)

## 7. 현재 단계 범위 (MVP)

### 포함
- 일정 생성/관리 (코트 수 변경 포함)
- 모임원 등록/관리
- 참여 상태 관리 (참여예정 → 대기중 ↔ 게임중 → 퇴장)
- 자동 게임 생성 (매칭 알고리즘)
- 수동 게임 생성
- 게임 종료 처리
- 실시간 대시보드
- 인증 없이 URL 접근으로 사용

### 미포함 (추후 확장)
- 회원가입/로그인
- 모임 생성 및 모임별 관리
- 단식(1v1) 지원
- 광고
- 인증/권한 관리

## 8. 프론트엔드 프레임워크 선정 근거

### Next.js (React) 선정 이유
1. **AI 코딩 최적화**: ChatGPT, Claude, Copilot 등 모든 AI 도구가 React/Next.js 코드를 가장 잘 생성함
2. **Firebase 통합**: `react-firebase-hooks` 등 검증된 라이브러리 존재
3. **자료 풍부**: 문제 발생 시 해결책을 찾기 가장 쉬움
4. **성능**: React 18의 Concurrent Features + Next.js의 최적화로 빠른 인터랙션 가능
5. **배포 간편**: Vercel 또는 Firebase Hosting에 간단히 배포 가능
